# swichcodex 产品需求文档

## 1. 产品定位

`swichcodex` 是一个 **macOS SwiftUI 原生桌面工具**，用于管理 Codex 账号、多开实例和本地会话。

目标用户：

- 在一台 Mac 上维护多个 Codex 账号的人
- 需要同时运行多个 Codex 实例的人
- 需要统一查看、整理、恢复 Codex 会话的人

产品边界：

- 只做 `Codex`
- 只做 `macOS`
- 一期不覆盖 `cockpit-tools` 里的其他平台能力

---

## 2. 一期目标

一期只做三块：

1. 账号总览
2. 多开实例
3. 会话管理

一期目标不是“大而全”，而是先做出一个稳定可用的原生版本。

---

## 3. 信息架构

主导航建议为三栏或顶部三 Tab：

1. `Accounts`
2. `Instances`
3. `Sessions`

通用要求：

- 支持浅色/深色模式
- 支持窗口缩放
- 首屏渲染快
- 错误提示明确
- 所有危险操作需要二次确认

---

## 4. Accounts 页面需求

## 4.1 页面目标

帮助用户统一管理 Codex 账号，支持导入、分组、刷新、切换。

## 4.2 页面结构

建议布局：

- 左侧：分组栏
- 右侧：账号列表区域
- 顶部工具栏：搜索、刷新、隐藏邮箱、添加账号

## 4.3 顶部工具栏

包含：

- `Add Account`
- `Refresh All`
- `Hide Email`
- 搜索框
- 布局切换：卡片 / 列表

按钮行为：

- `Add Account`：打开添加账号弹窗
- `Refresh All`：逐个刷新当前可见账号
- `Hide Email`：切换邮箱显示为脱敏或完整
- 搜索：按邮箱、显示名、标签过滤

## 4.4 左侧分组栏

展示：

- `All`
- `Ungrouped`
- 用户自定义分组

支持行为：

- 新建分组
- 重命名分组
- 删除分组
- 查看组内账号数
- 切换分组过滤

## 4.5 账号卡片

每张卡片展示：

- 显示名
- 邮箱或脱敏邮箱
- 账号类型
- plan 信息
- 配额摘要
- 最后刷新时间
- 标签
- 是否当前账号

每张卡片操作：

- `Switch`
- `Refresh`
- `Edit`
- `Delete`
- `Move to Group`

## 4.6 添加账号

一期支持三种方式：

1. 从本地导入当前 `~/.codex/auth.json`
2. 手动粘贴 token
3. 手动输入 API Key

暂不要求一期支持完整 OAuth 浏览器登录。

## 4.7 切换账号

切换后要求：

- 当前账号高亮
- 写入默认 `~/.codex/auth.json`
- 如默认实例跟随当前账号，则同步更新绑定关系
- 给出成功/失败反馈

## 4.8 刷新逻辑

支持：

- 刷新单账号
- 刷新全部账号

刷新结果要更新：

- profile 信息
- plan 信息
- 配额信息
- 最后刷新时间

## 4.9 异常提示

典型错误：

- 本地 `auth.json` 不存在
- token 无效
- API Key 格式错误
- 网络请求失败
- 当前账号切换失败

要求：

- 错误提示可读
- 不直接暴露技术堆栈细节
- 支持重试

---

## 5. Instances 页面需求

## 5.1 页面目标

帮助用户创建和管理多个独立的 Codex 实例。

## 5.2 页面结构

建议布局：

- 主区域为实例列表
- 右侧或底部为实例详情
- 顶部工具栏放新增、刷新、关闭全部

## 5.3 实例列表字段

每条实例展示：

- 实例名
- 是否默认实例
- 绑定账号
- 实例目录
- 启动模式
- 运行状态
- 最后启动时间

## 5.4 实例操作

每条实例支持：

- `Start`
- `Stop`
- `Open Window`
- `Refresh Status`
- `Edit`
- `Delete`

页面级操作：

- `Create Instance`
- `Close All`

## 5.5 创建实例

创建实例弹窗字段：

- 实例名
- 实例目录
- 初始化方式
- 绑定账号
- 启动模式
- 工作目录
- 附加参数

一期初始化方式建议支持：

1. 空白目录
2. 使用现有目录

是否复制来源实例可放到二期。

## 5.6 绑定账号

要求：

- 实例可以绑定一个账号
- 默认实例可以选择“跟随当前账号”
- 启动前自动将绑定账号写入目标实例目录

## 5.7 启动模式

一期建议支持：

- `App`

二期再扩展：

- `CLI`

因为一期重点是先把原生工具跑通，减少终端分支带来的复杂度。

## 5.8 状态刷新

状态刷新包括：

- 进程是否存在
- PID 是否仍有效
- 目录是否已初始化
- 绑定信息是否完整

## 5.9 删除实例

删除前必须提示：

- 是否删除实例记录
- 是否删除实例目录

默认建议：

- 删除实例记录
- 保留实例目录

避免误删用户数据。

---

## 6. Sessions 页面需求

## 6.1 页面目标

帮助用户聚合查看所有实例中的 Codex 会话，并支持删除与恢复。

## 6.2 页面结构

建议布局：

- 左侧：按 cwd 分组的项目列表
- 右侧：会话列表
- 顶部工具栏：刷新、查看 Trash

## 6.3 会话列表字段

每条会话展示：

- 会话标题
- `sessionId`
- `cwd`
- 更新时间
- 所在实例数
- token 统计

可展开信息：

- 分布在哪些实例
- 对应实例是否在运行

## 6.4 会话操作

一期支持：

- `Refresh`
- `Copy Session ID`
- `Move to Trash`
- `Open Trash`

Trash 页面支持：

- 列出已删除会话
- 恢复会话

## 6.5 分组方式

默认按 `cwd` 分组。

分组行为：

- 默认折叠
- 记住最近展开状态
- 支持搜索过滤

## 6.6 token 统计

从 rollout 数据中读取：

- input tokens
- output tokens
- total tokens

要求：

- 首次按需加载
- 加载失败不阻塞主列表

## 6.7 删除与恢复

删除原则：

- 只做软删除
- 进入应用自己的 Trash 目录
- 删除前做必要备份

恢复原则：

- 只恢复存在 manifest 的记录
- 恢复前检查目标实例数据是否冲突

## 6.8 风险提示

如果目标实例正在运行，要提示：

- 删除/恢复后界面未必即时生效
- 可能需要重启实例才能看到结果

---

## 7. 通用交互要求

## 7.1 加载态

每个页面都需要：

- 首屏 skeleton 或 progress view
- 局部操作 loading
- 长耗时任务反馈

## 7.2 成功反馈

建议使用：

- 顶部轻提示
- 行内成功状态

避免过度弹窗。

## 7.3 危险操作确认

以下操作必须二次确认：

- 删除账号
- 删除实例
- 删除会话
- 恢复会话覆盖风险

## 7.4 空状态

需要为空状态提供清晰引导：

- 无账号时提示先导入账号
- 无实例时提示创建实例
- 无会话时提示刷新或启动实例后再查看

---

## 8. 非功能需求

要求：

- macOS 13+
- 原生滚动、原生快捷键、原生窗口体验
- 不阻塞主线程
- 文件写入必须原子化
- 出错时不破坏用户现有 Codex 数据

---

## 9. 一期验收标准

满足以下条件可视为一期可交付：

1. 可以导入并展示多个 Codex 账号
2. 可以切换默认账号
3. 可以为账号分组、隐藏邮箱、刷新账号信息
4. 可以创建多个实例并绑定账号
5. 可以启动、停止、定位实例窗口
6. 可以聚合展示所有实例会话
7. 可以将会话移入 Trash 并恢复
8. 危险操作都有确认和失败提示

---

## 10. 二期候选需求

- 完整 OAuth 登录流程
- CLI 启动模式
- 跨实例线程同步
- 会话可见性修复
- 菜单栏模式
- Keychain 存储
- 更完整的过滤和排序能力
